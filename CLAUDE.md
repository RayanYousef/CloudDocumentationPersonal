## What this repository is

This is a **documentation project**, not an application. It documents how to set up a Unity → GitHub Actions cloud build pipeline (Android APK/AAB, iOS IPA → TestFlight). The "code" it contains (Unity C# editor scripts, GitHub Actions workflow YAML) is *documented subject matter / downloadable artifacts*, not a buildable program. The actual buildable deliverable is the Docusaurus site in `website/`.

## Commands

All site commands run from the `website/` directory (npm is the convention used by the deploy workflow and the in-site User Guide):

```bash
cd website
npm install          # or `npm ci` for a clean, lockfile-exact install
npm run start        # local dev server with hot reload at http://localhost:3000
npm run build        # static build into website/build/
npm run serve        # serve the production build locally
npm run clear        # clear the Docusaurus cache (use when build behaves oddly)
```

There are no tests or linters configured. Requires Node ≥ 20 (Docusaurus 3.9, React 19).

## Deployment

`.github/workflows/deploy-docs.yml` auto-deploys to GitHub Pages on every push to `main` that touches `website/**`. It runs `npm ci` + `npm run build` and publishes `website/build` to the `gh-pages` branch. Site config: `url=https://RayanYousef.github.io`, `baseUrl=/CloudDocumentationPersonal/` — any absolute in-site link must include that base path.

## Architecture: content lives in three places (the key gotcha)

The same content is duplicated across three locations. **When you change a Unity script or a workflow file, you must update all relevant copies or the published site will drift from reality:**

1. **`docs/` (repo root)** — the original GitHub-repo-facing guides plus the *real* artifact files:
   - `docs/scripts/*.cs` — the actual Unity editor scripts (source of truth)
   - `.github/workflows/*.yml` — the actual CI/CD workflows (source of truth), e.g. `main-controller.yml`, `android-*.yml`, `ios-ipa-testflight.yml`
   - `docs/CloudBuild/` — `VersioningSettings.json` and localized `WhatsNew/` release notes
   - `docs/images/`
   - The root `README.md` is the GitHub landing page and links into `docs/`.

2. **`website/docs/`** — the published Docusaurus rendering of those guides, reorganized into sidebar categories: `getting-started/`, `platforms/{android,ios}/`, `reference/`, `troubleshooting/`, `GUIDE/` (the end-user "how to edit this site" guide).

3. **`website/static/downloads/{scripts,workflows}/`** — copies of the `.cs` and `.yml` files, served as download links. `website/docs/reference/scripts.md` and `workflows.md` **both embed the code inline as fenced blocks AND link to these download copies** (paths like `/CloudDocumentationPersonal/downloads/scripts/...`). So a single script change can require editing: the source file in `docs/scripts/`, the copy in `website/static/downloads/scripts/`, and the inline code block in `website/docs/reference/scripts.md`.

`misc/OriginalDocumentation/` is an archived snapshot of the original docs — not part of the live site.

## Authoring conventions for the Docusaurus site

- The sidebar is **auto-generated** from the folder structure under `website/docs/` (`sidebars.js` uses `autogenerated`). Folder = sidebar category; file = page. Docs are served at the site root (`routeBasePath: '/'`); the blog is disabled.
- Each page uses front matter (`title`, `sidebar_position`, `description`; `slug: /` marks the homepage at `getting-started/intro.md`).
- Control a category's label/order/collapse via `_category_.json` in that folder. `sidebar_position` orders pages within a folder.
- Numeric filename prefixes (e.g. `GUIDE/01-using-the-site.md`) also drive ordering.
- Images go in `website/static/img/` and are referenced with absolute paths like `/img/foo.png`.
- The in-site `website/docs/GUIDE/` pages document these same conventions for non-technical content editors — keep them in sync if you change how the site is structured.
